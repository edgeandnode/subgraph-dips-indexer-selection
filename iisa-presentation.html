V1 Architecture and Scoring Design

The Graph · DIPs Indexer Selection

Architecture

How It Works

A scoring service replays 28 days of real gateway query traffic from Redpanda daily, combines it with on-chain stake data from the Graph Network subgraph and pricing fetched directly from each indexer's /dips/info endpoint, then computes a quality score per indexer and writes the results to a shared volume. A separate HTTP service loads those pre-computed scores and serves selection requests on demand -- selection is fast because all the heavy computation happened in the background.

Core Logic

Scoring and Normalisation

Each indexer is scored across six dimensions, normalised to [0, 1] where 1 is best, then combined via weighted average. Missing metrics are excluded from both numerator and denominator -- weights redistribute across available dimensions rather than penalising indexers for missing data.

How normalisation works

Uptime and Success Rate

Piecewise linear scaling: indexers below 97% of the best performer's value score zero. Only the top 3% band maps linearly from 0 to 1. Most indexers cluster in the high 90s, so differentiation happens at the top -- 98.5% of best scores 0.5, while 96% scores 0.

Price

Normalised as 1 - (price / ceiling). When dipper provides a budget, that is the ceiling. Otherwise, ceiling falls back to the observed max among surviving candidates (post hard-filter). Floor is always 0 GRT. Indexers above budget are filtered out entirely before scoring.

Latency

Scored via linear regression controlling for distance, deployment, and query characteristics. Uses an upper-bound confidence interval (coeff + 1.5 × SE), so indexers with few samples or inconsistent performance are scored conservatively.

Stake-to-Fees

(staked - locked) / total_fees. Indexers with zero query fees normalise to 1.0 (maximum capacity headroom). Higher ratio means more economic security per unit of fee revenue.

Selection

How It Selects

When dipper sends a POST /select-indexers request with a deployment ID, target group size, chain, and optional budget, the service first filters candidates by DIP availability, chain support, and budget, then normalises metrics and computes weighted scores for the surviving set.

Group assignment

ConditionActionDetails
Under-staffed Add indexers Adds the highest-scoring eligible candidate. Repeats until target size reached.
Over-staffed Remove worst Removes the lowest-scoring indexer. Repeats until at target.
At target Replace check Only indexers scoring below 0.15 are candidates for replacement. A swap requires the candidate to score at least 0.50 higher -- preventing churn from marginal improvements.

Decentralisation

Groups of 2+ indexers prefer at least 2 unique ASN organisations and 2 unique geographic locations. This is a soft constraint: the algorithm tries diversity-satisfying candidates first, but falls back to the overall best scorer if the constraint can't be met.

Load spreading

When dipper passes expected fees from already-accepted agreements (optimistic_dips_fees), these are added to fee totals before recomputing stake_to_fees. This naturally deprioritises indexers that already have agreements, spreading load across the network.

Operations

Resilience

The system degrades gracefully through three operating modes.

Full

All data sources available. Complete pipeline with latency regression, GeoIP, and real pricing.

Partial

GeoIP unavailable. Latency set to neutral (0.5). Uptime, success rate, and stake-to-fees still from real data.

Degraded

Redpanda also unavailable. All quality metrics equal. Differentiation from real pricing only.

Philosophy

V1 Design Rationale

Orthogonal dimensions, configurable weights

Six independent axes that can be tuned individually. Adding a new dimension means adding one column and one weight, not restructuring the algorithm.

Regression-based latency

Controls for distance, deployment, and query effects instead of conflating fast-nearby indexers with slow ones. The confidence interval penalty handles cold-start conservatively.

Soft decentralisation

Hard constraints make selection infeasible when the candidate pool is small. Prefer diversity, but never block a selection entirely. Tighten as the indexer population grows.

Conservative replacement

A 0.50 score improvement threshold prioritises stability. Frequent churn disrupts service and forces indexers to repeatedly sync and accept agreements.
